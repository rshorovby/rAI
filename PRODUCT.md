# RallyAI — продуктовое понимание

Документ зафиксирован по итогам product grill (июль 2026). Источник истины для scope, монетизации и must-have до первых платящих.

---

## 1. Продуктовая идентичность

**MVP сейчас → коммерческий продукт в горизонте 3–6 месяцев.**

Не личная игрушка и не сразу SaaS на тысячи пользователей. Цель — публичная бета с инфраструктурой, готовой к монетизации.

---

## 2. Кто платит

**B2C:** игрок-любитель или продвинутый любитель.

| Сегмент | Роль |
|---------|------|
| **Primary** | Игрок с живым тренером — AI-разбор между занятиями |
| **Opportunistic** | Игрок без тренера — принимаем, не оптимизируем GTM |

B2B (тренеры, клубы) — **вне scope** на текущем этапе.

**Позиционирование:** «домашняя работа между уроками», не «тренер в кармане».

---

## 3. Критерии готовности к оплате

Включать платёж только после **всех четырёх** условий:

| Блок | Порог |
|------|-------|
| **Качество** | ≥15 разборов с фидбеком; ≥60% «полезно / применимо»; 0 критичных кейсов по травмам из профиля |
| **Удержание** | ≥30% пользователей с 1-м разбором отправляют 2-е видео за 14 дней |
| **Техника** | `analysis_success / video_sent ≥ 90%` за 30 дней; известна себестоимость разбора; маржа ≥50% |
| **WTP** | ≥5 человек явно готовы платить (до интеграции платежей, вручную) |

**Не требовать до оплаты:** 1000 пользователей, идеальный промпт, веб/мобильное приложение.

---

## 4. Модель монетизации

| Элемент | Решение |
|---------|---------|
| Free tier | 1 полный разбор бесплатно |
| Paid | Подписка **4 разбора/мес** (сброс лимита ежемесячно) |
| Апселл | Разовый пакет **+2 разбора** при исчерпании лимита |
| Follow-up в сессии | **Не считается** в лимит |

**Платежи:** сначала ручная выдача доступа → ЮKassa / Telegram Stars после 5 ручных платящих.

## 6. Границы MVP

### Не делаем до первых 5–10 платящих

- Веб / iOS / Android
- B2B (кабинет тренера, клубы)
- Графики прогресса, сравнение видео до/после
- Библиотека упражнений, социальные фичи, рефералка
- PostgreSQL, Redis, микросервисы
- Amplitude, A/B-тесты

### Отложено осознанно

- ЮKassa — после 5 ручных оплат

## 7. Definition of done — первый разбор

После первого разбора игрок должен:

1. **Назвать одну главную ошибку** своими словами.
2. **Иметь одно конкретное домашнее задание** из «Топ-3 приоритетов».
3. **Знать, что снять на следующее видео** — удар, ракурс, 10–20 сек.
4. **Вернуться через 7–14 дней** с новым роликом.

**Формулировка успеха:**

> «Я понял, над чем работать до следующего занятия с тренером, и знаю, что снять, чтобы проверить прогресс.»

**Следствия для продукта:**

- CTA в конце отчёта / `followup_hint`: «Сними [удар] с [ракурса] через неделю».
- Напоминание через 7 дней — с отсылкой к заданию, не generic.
- Вопрос фидбека: «Понятно, что делать на тренировке?»

---

## 8. Резюме в одном предложении

**RallyAI** — Telegram-бот для русскоязычных любителей тенниса с тренером: один бесплатный AI-разбор видео, далее 4 разбора в месяц; ценность — понятная домашняя работа между занятиями и возврат с новым роликом через неделю.

## 9. Must-have: конкретные задачи

Порядок — по зависимостям. ID задач (`MH-*`) можно использовать в коммитах и issues.

### MH-1 · Фидбек после разбора

**Зачем:** критерий качества ≥60% «полезно».

| Поле | Значение |
|------|----------|
| Файлы | `bot.py`, `analytics.py`, `storage.py`, `i18n.py`, `tests/test_analytics.py`, `tests/test_bot.py` |
| Зависимости | нет |

**Шаги:**

1. Добавить события `feedback_positive`, `feedback_negative` в `analytics.py` → `ALL_EVENT_TYPES`.
2. После успешного разбора в `_run_video_analysis` отправить inline-клавиатуру: 👍 Полезно / 👎 Не помогло + опционально «Понятно, что делать на тренировке?» (да/нет) вторым рядом или follow-up после 👍.
3. `CallbackQueryHandler` с pattern `^fb:` → `storage.log_event`, благодарность пользователю, убрать кнопки (`edit_message_reply_markup`).
4. Строки в `i18n.py` (ru/en): подписи кнопок, «спасибо за отзыв».
5. В `format_analytics_report` — блок «Фидбек»: count positive, negative, % positive от всех feedback-событий.
6. Тесты: handler логирует событие; формат отчёта включает фидбек.

**Критерий готовности:** после разбора пользователь видит кнопки; события попадают в `/stats` и `stats_report.py`.

---

### MH-2 · CTA «что снять» в отчёте и follow-up

**Зачем:** definition of done п.3–4.

| Поле | Значение |
|------|----------|
| Файлы | `prompts.py`, `i18n.py`, `bot.py`, `tests/test_prompts.py` |
| Зависимости | нет (параллельно с MH-1) |

**Шаги:**

1. В `USER_PROMPT_RU` / `USER_PROMPT_EN` добавить обязательную секцию `## Следующее видео` (или финальный абзац): один удар, ракурс, длительность 10–20 сек, через ~7 дней.
2. В system/follow-up prompt: если секции нет в отчёте — модель должна напомнить при первом вопросе.
3. Заменить `followup_hint` в `i18n.py` на текст с отсылкой к домашнему заданию и следующему ролику (не только «задавайте вопросы»).
4. При сохранении сессии в `storage.save_session` извлекать и сохранять `next_video` (новое поле `next_video TEXT` в `player_sessions` или парсинг из отчёта).
5. Тест: промпт содержит требование секции; `_extract_report_sections` / новый extractor возвращает `next_video`.

**Критерий готовности:** каждый отчёт заканчивается конкретным заданием «что снять»; текст виден в Telegram после разбора.

### MH-3 · Копирайт `/about`, welcome, онбординг

**Зачем:** позиционирование primary-сегмента «между занятиями с тренером».

| Поле | Значение |
|------|----------|
| Файлы | `i18n.py`, `tests/test_i18n.py` |
| Зависимости | нет |

**Шаги:**

1. Обновить `welcome`, `bot_description`, `bot_short_description`, `about` (ru): явно «между тренировками с тренером», «домашняя работа», не «замена тренера».
2. Обновить `ob_intro`, `ob_complete`, `ob_skip_warning` (ru): связь с занятиями у тренера.
3. Зеркально обновить EN-строки (passive market).
4. Тест: ключевые фразы присутствуют в ru `about` и `welcome`.

**Критерий готовности:** новый пользователь по `/about` и онбордингу понимает use case «между уроками».

### MH-4 · Вернуть быстрые кнопки после разбора

**Зачем:** ценность follow-up без нового видео; код уже есть (`_quick_questions_keyboard`, `handle_quick_question`).

| Поле | Значение |
|------|----------|
| Файлы | `bot.py`, `tests/test_bot.py` |
| Зависимости | MH-1 (кнопки фидбека и quick questions не конфликтуют — двумя сообщениями или одной клавиатурой) |

**Шаги:**

1. После отправки чанков отчёта в `_run_video_analysis` отправить сообщение с `_quick_questions_keyboard(lang)` (отдельно от фидбека MH-1).
2. Убедиться, что `CallbackQueryHandler` для `^q:` зарегистрирован (уже есть в `build_application`).
3. Follow-up через quick question **не** списывает лимит разборов (зафиксировать в коде/комментарии product rule; тест при наличии MH-5).
4. Тест: после мок-анализа бот отправляет inline-кнопки «Разбор главной ошибки» / «3 упражнения».

**Критерий готовности:** кнопки видны после каждого успешного разбора; callback вызывает `_process_followup`.

---

### MH-5 · Лимит разборов (freemium)

**Зачем:** модель 1 бесплатный + 4/мес; блокировка без оплаты.

| Поле | Значение |
|------|----------|
| Файлы | `storage.py`, `bot.py`, `i18n.py`, `tests/test_storage.py`, `tests/test_bot.py` |
| Зависимости | нет |

**Шаги:**

1. Расширить схему SQLite — таблица `user_quotas`:
   - `user_id`, `plan` (`free` | `paid`), `analyses_used`, `period_start` (ISO date), `bonus_credits` (для пакета +2).
2. API в `storage.py`: `get_quota(user_id)`, `can_analyze(user_id) -> bool`, `consume_analysis(user_id)`, `reset_period_if_needed(user_id)`.
3. Логика: новый пользователь → 1 бесплатный разбор (`analyses_used` после первого success = 1, дальше block); paid → 4 в календарном месяце, сброс при смене месяца.
4. В `handle_video` / `_run_video_analysis` перед анализом проверять `can_analyze`; при отказе — сообщение `quota_exhausted` (ru/en) с CTA «напишите администратору» / позже оплата.
5. Follow-up (текст и quick buttons) **не** вызывает `consume_analysis`.
6. Тесты: free user блокируется после 1-го success; paid user — после 4-го в периоде.

**Критерий готовности:** второе видео от free-пользователя без admin-grant получает отказ с понятным сообщением.

---

### MH-6 · Ручная выдача paid-доступа (админ)

**Зачем:** первые 5 платящих без ЮKassa.

| Поле | Значение |
|------|----------|
| Файлы | `bot.py`, `storage.py`, `config.py`, `.env.example`, `tests/test_bot.py` |
| Зависимости | MH-5 |

**Шаги:**

1. Команда `/grant <user_id> [bonus]` — только для `ADMIN_USER_IDS` (аналог `/stats`).
2. `storage.grant_paid(user_id, bonus_credits=0)` → `plan=paid`, сброс `analyses_used`, `period_start=сегодня`.
3. Команда `/revoke <user_id>` — вернуть в `free` (для отладки).
4. Логировать событие `admin_grant` в `events`.
5. Документировать в `AGENTS.md` или комментарии в `PRODUCT.md` (уже здесь): workflow ручной оплаты.
6. Тест: non-admin получает отказ; admin grant открывает 4 разбора.

**Критерий готовности:** админ может выдать paid по Telegram user_id без деплоя нового кода.

---

### MH-7 · Себестоимость разбора

**Зачем:** критерий маржи ≥50%.

| Поле | Значение |
|------|----------|
| Файлы | `analyzer.py`, `storage.py`, `analytics.py`, `bot.py`, `stats_report.py` |
| Зависимости | нет (параллельно) |

**Шаги:**

1. В `VideoAnalyzer.analyze` и `.chat` читать `response.usage_metadata` (input/output tokens) из `GenerateContentResponse`.
2. Логировать в struct log: `user_id`, `kind=analyze|chat`, `model`, tokens, `video_size_bytes` (из Telegram metadata).
3. Сохранять в `events` с типом `api_usage` и payload JSON `{input_tokens, output_tokens, kind}` или отдельная таблица `usage_log`.
4. В `format_analytics_report` — блок «API за 30 дней»: сумма tokens, среднее tokens/разбор, оценка cost в USD по прайсу `GEMINI_MODEL` (константа в `config.py` или `analytics.py`, обновлять вручную).
5. Опционально: `stats_report.py --cost` или секция в существующем отчёте.
6. Тест: мок response с `usage_metadata` → событие/запись создаётся.

**Критерий готовности:** после 10 разборов админ видит среднюю себестоимость одного analyze в `/stats`.

---

### MH-8 · Напоминание «пришли видео» через 7 дней

**Зачем:** retention ≥30% (2-е видео за 14 дней).

| Поле | Значение |
|------|----------|
| Файлы | `bot.py`, `storage.py`, `main.py`, `i18n.py`, `deploy/rallyai-reminder.cron` (новый), `tests/test_storage.py` |
| Зависимости | MH-2 (`next_video` для персонализации) |

**Шаги:**

1. Поле `last_analysis_at` (ISO datetime) в `users` или `player_sessions` — обновлять в `save_session`.
2. `storage.get_users_for_reminder(days=7) -> list[user_id, lang, next_video]` — пользователи, у которых последний success был ровно 7±0 дней назад и не было нового видео после.
3. Флаг `reminder_sent_at` на пользователя или событие `reminder_sent` чтобы не слать повторно в том же цикле.
4. Скрипт `remind.py` (или `python -m reminders`): загружает settings, для каждого кандидата шлёт через Bot API сообщение с текстом из `i18n` (`reminder_7d` + подстановка `next_video`).
5. Cron на сервере: `0 10 * * *` (10:00 UTC или MSK — зафиксировать) `deploy/rallyai-reminder.cron`; отдельный от systemd-сервиса бота.
6. Событие `reminder_sent` в analytics; в `/stats` — count reminders / return after reminder (опционально phase 2).
7. Тесты: `get_users_for_reminder` с фиксированными датами в SQLite.

**Критерий готовности:** пользователь с разбором 7 дней назад получает персональное напоминание с заданием из отчёта; повторно в тот же день не получает.

## 11. Метрики для проверки критериев §3

| Метрика | Где считать | Порог |
|---------|-------------|-------|
| % полезно | `events` type `feedback_positive` / all feedback | ≥60% |
| Retention 14d | SQL: users с 2+ `analysis_success` за 14d / users с 1+ | ≥30% |
| Success rate | `analysis_success` / `video_sent` за 30d | ≥90% |
| Себестоимость | `/stats` блок API (MH-7) | маржа ≥50% при цене X |
| WTP | вне бота, ручной опрос | ≥5 человек |

Цену X (руб/мес за 4 разбора) зафиксировать отдельно после первых 15 фидбеков — в этом документе не задана намеренно.
